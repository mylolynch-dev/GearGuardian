--------------------------------------------------------------------------------
WHAT IS GEAR GUARDIAN?
--------------------------------------------------------------------------------

Gear Guardian is a portable anti-theft alarm device designed to protect camera
bags, backpacks, and gear cases. The firmware runs on an ESP32-S3 DevKitC-1
using Zephyr RTOS — a professional-grade real-time operating system used in
commercial embedded products.

When a bag is sealed and the device is armed, it monitors for physical
disturbances using an ICM-20948 9-axis IMU (accelerometer + gyroscope). If the
bag is opened or moved aggressively, a buzzer alarm fires, an OLED display
updates, and a CSV log entry is written to a microSD card.

The project was built from scratch with zero existing code as a portfolio
firmware project — every register, every thread, every state transition written
by hand.

--------------------------------------------------------------------------------
HARDWARE COMPONENTS
--------------------------------------------------------------------------------

Core:
  - ESP32-S3 DevKitC-1 (Xtensa LX7 dual-core, 512 KB SRAM, Wi-Fi + BLE)
  - Zephyr RTOS — the same OS used in medical devices, automotive, and IoT

Sensors / Inputs:
  - ICM-20948 IMU — accelerometer + gyroscope over I2C
    (I2C address 0x69, WHO_AM_I register returns 0xEA)
  - Magnetic reed switch — detects bag open/close via GPIO interrupt
  - Mode button — boot-time GPIO read to select Normal vs Diagnostic mode

Outputs:
  - SSD1306 128x64 OLED display — status screens via SPI
  - Active buzzer — alarm patterns, chirps, SOS pattern via GPIO
  - Status LED — onboard RGB LED on GPIO48

Storage:
  - MicroSD card over SPI — FAT32 filesystem for CSV event logs
  - Internal NVS flash — non-volatile boot counter and fault persistence

Build Tools:
  - West (Zephyr's meta-build tool)
  - CMake + Ninja
  - Zephyr SDK 0.17.4 (xtensa-espressif_esp32s3 toolchain)
  - Python 3.13, esptool for flashing

--------------------------------------------------------------------------------
THE ORIGINAL PLAN — 6 PHASES
--------------------------------------------------------------------------------

The project was designed in six phases, moving from compile-clean scaffold to
a fully tuned and hardened firmware:

  Phase 1 — Full scaffold: all headers, driver stubs, build system, docs
  Phase 2 — Small drivers: reed switch (ISR + debounce), buzzer, mode button
  Phase 3 — ICM-20948 driver from datasheet registers (I2C, bank switching)
  Phase 4 — Normal mode state machine + event dispatcher + motion classifier
  Phase 5 — Diagnostic mode, safe mode, boot metadata with NVS
  Phase 6 — Watchdog, stack audit, threshold tuning, final polish

PHASE 1 — COMPLETE: FULL PROJECT SCAFFOLD

STATUS: Done. Clean build achieved. Zero errors, zero warnings.

Everything was written from scratch in one pass:

--- APPLICATION HEADERS (app/include/) ---

app_faults.h
  - Defines fault bitmask enum using BIT(n) macros
  - Faults: IMU_INIT, SD_WRITE, NVS_CORRUPT, WATCHDOG, ASSERT, SENSOR_TIMEOUT
  - APP_FATAL_FAULT_MASK — faults that force the system into safe mode
  - Inline helper: app_fault_is_fatal(flags)

app_modes.h
  - enum app_mode: BOOT, NORMAL, DIAGNOSTIC, SAFE
  - enum app_normal_substate: DISARMED, ARMING, ARMED, ALARM, COOLDOWN

app_config.h
  - Single header for ALL project-wide constants (no magic numbers anywhere)
  - I2C address for ICM-20948: 0x69
  - Arming delay: 5000ms, cooldown: 30000ms, sample rate: 100ms
  - Stack sizes and thread priorities for all 5 threads
  - SD log filename: "/SD:/gg_log.csv"
  - Motion thresholds (tunable post-hardware testing)
  - OLED dimensions: 128x64

boot_metadata.h
  - Defines struct boot_meta with magic number 0x6561723U
  - Tracks: boot count, consecutive failures, cumulative fault flags, reset cause
  - API: boot_meta_init(), boot_meta_load(), boot_meta_save()
  - boot_meta_record_clean_boot(), boot_meta_record_failed_boot(faults)

app_events.h
  - Defines all events the system can generate:
    EVT_REED_OPEN, EVT_REED_CLOSE
    EVT_IMU_SAMPLE_READY
    EVT_MOTION_DETECTED, EVT_MOTION_CLEARED
    EVT_ALARM_SILENCE
    EVT_COOLDOWN_EXPIRE
    EVT_MODE_CHANGE
    EVT_LOG_RECORD
    EVT_FAULT_DETECTED
  - struct app_event with union payload capped at 56 bytes total
    (IMU branch: 6x int16 + int64 = 20B; log branch: char[48])
  - extern declaration for app_event_queue (defined in event_dispatcher.c)

app_state.h
  - struct app_state: holds current mode, substate, reed state, alarm active flag,
    last IMU sample, and copies of boot metadata counters
  - extern g_app_state + g_app_state_mutex (mutex-protected shared state)
  - Inline accessor helpers for thread-safe reads

--- APPLICATION SOURCE FILES (app/src/) ---

main.c
  - Calls startup_run() then goes idle
  - All threads are started via SYS_INIT at APPLICATION priority

startup.c
  - Boot coordinator: reads mode button GPIO synchronously before threads start
  - Initializes all drivers in dependency order
  - Decides whether to enter normal, diagnostic, or safe mode
  - Reads hardware reset cause via hwinfo_get_reset_cause()

mode_manager.c
  - Central dispatcher for entering/exiting modes
  - Calls mode-specific enter() and exit() functions

normal_mode.c / diag_mode.c / safe_mode.c
  - Each mode has enter() and exit() functions
  - Normal mode: arms monitoring, starts full thread set
  - Diagnostic mode: streams IMU data to OLED, tests all hardware
  - Safe mode: minimal display + buzzer SOS, blocks further arming

state_machine.c
  - Owns g_app_state and g_app_state_mutex
  - Implements substate transitions for normal mode:
    DISARMED → ARMING (reed close)
    ARMING → ARMED (timer expire, arming delay elapsed)
    ARMING → DISARMED (reed open, abort)
    ARMED → ALARM (reed open or motion detected)
    ALARM → COOLDOWN (silence command or timeout)
    COOLDOWN → DISARMED (cooldown timer expire)

event_dispatcher.c
  - Defines K_MSGQ_DEFINE(app_event_queue, sizeof(app_event_t), 16, 4)
  - Highest-priority thread (p2): dequeues events and routes them
  - Calls state_machine, motion_classifier, logger based on event type
  - ISR-safe: reed_switch and sensor_thread post events via k_msgq_put

sensor_thread.c
  - Reads ICM-20948 at APP_SAMPLE_INTERVAL_MS (100ms)
  - Posts EVT_IMU_SAMPLE_READY with raw int16 accel + gyro values

motion_classifier.c
  - Sliding window ring buffer of 8 samples
  - Computes delta from rolling mean
  - Posts EVT_MOTION_DETECTED when threshold exceeded
  - Posts EVT_MOTION_CLEARED when below clear threshold

fault_manager.c
  - Manages global fault bitmask
  - app_fault_set(id): ORs in fault bit, saves to NVS via boot_metadata
  - app_fault_is_fatal() check → triggers safe_mode_enter()

boot_metadata.c
  - Currently a RAM-only stub (Phase 5 will wire to NVS)
  - Tracks boot count and consecutive failure count in memory

logger_service.c
  - Dedicated thread (p5) with its own internal message queue
  - Receives struct log_record: {timestamp_ms, mode, substate, event_type,
    fault_flags, summary[48]}
  - Calls sdlog_write() per record; on failure sets FAULT_SD_WRITE

ui_service.c
  - Thread (p6) blocked on k_sem_take(&ui_update_sem, K_FOREVER)
  - Reads state under mutex, calls appropriate oled_screen_*() function
  - Screen functions: boot, disarmed, arming, armed, alarm, diagnostic,
    safe, imu_data

alarm_service.c
  - Thread (p5) that manages buzzer patterns
  - Receives alarm on/off commands
  - Runs buzzer_alarm_pattern() (3x 500ms on / 200ms off) in a loop

--- DRIVERS (drivers/) ---

icm20948_regs.h
  - Pure #define register map (no code, no types)
  - REG_BANK_SEL = 0x7F (accessible from all banks)
  - Bank 0: WHO_AM_I = 0x00, PWR_MGMT_1 = 0x06, ACCEL_XOUT_H = 0x2D
  - Bank 2: GYRO_CONFIG_1 = 0x01, ACCEL_CONFIG = 0x14, sample rate divisors
  - All bitfield masks defined (e.g. PWR_MGMT_1_DEVICE_RESET = BIT(7))

icm20948.c (stubbed, Phase 3 will implement)
  - Will implement bank switching, WHO_AM_I check, reset sequence,
    config register writes, and 14-byte burst read

reed_switch.c
  - GPIO interrupt (EDGE_TO_ACTIVE and EDGE_TO_INACTIVE)
  - ISR submits k_work_delayable with 5ms debounce
  - Work handler reads pin, posts EVT_REED_OPEN or EVT_REED_CLOSE

buzzer.c
  - buzzer_on() / buzzer_off(): direct GPIO
  - buzzer_chirp(ms): on + k_sleep + off
  - buzzer_alarm_pattern(): 3 x (500ms on / 200ms off)
  - buzzer_diag_pattern(): 2 short chirps
  - buzzer_sos_pattern(): S-O-S in morse timing

oled.c
  - Wraps Zephyr display API (guarded with #ifdef CONFIG_DISPLAY)
  - Local 1KB framebuffer for SSD1306 128x64 monochrome
  - oled_draw_text(), oled_draw_hline(), oled_flush()
  - Screen renderers: boot, disarmed, arming, armed, alarm, diagnostic,
    safe, imu_data — all implemented, call draw_text with layout coordinates

sdlog.c
  - Entire FatFS implementation guarded with #ifdef CONFIG_FILE_SYSTEM
  - When SD disabled: stubs return -ENOTSUP so rest of code compiles clean
  - When enabled: FATFS mount at "/SD:", CSV header on first open,
    fs_write per record, fs_sync on flush

--- BUILD SYSTEM ---

CMakeLists.txt
  - Flat single-level build: all sources listed explicitly
  - Key fix: set(DTC_OVERLAY_FILE ...) before find_package(Zephyr)
    Zephyr 4.x with qualified board target esp32s3_devkitc/esp32s3/procpu
    does NOT auto-discover overlays named only after the board root name
  - Include paths: app/include + all driver subdirs (no path prefixes in code)

prj.conf
  - Core: MAIN_STACK_SIZE=2048, HEAP_MEM_POOL_SIZE=8192
  - Logging: CONFIG_LOG=y, LOG_DEFAULT_LEVEL=3, CONFIG_PRINTK=y
  - Peripherals enabled: CONFIG_I2C=y, CONFIG_SPI=y, CONFIG_GPIO=y
  - NVS stack: CONFIG_NVS=y, CONFIG_FLASH=y, CONFIG_FLASH_MAP=y
  - CONFIG_HWINFO=y for reset cause detection
  - CONFIG_FPU=y, CONFIG_FPU_SHARING=y (hardware FPU on Xtensa LX7)
  - CONFIG_STACK_SENTINEL=y for stack overflow detection during dev
  - OLED commented out: no DT node until hardware is wired
  - SD/filesystem commented out: no disk driver until SD card is wired

boards/esp32s3_devkitc.overlay
  - &i2c0: ICM-20948 at 0x69 with compatible = "vnd,i2c-device" (no upstream
    Zephyr binding for ICM-20948 exists yet; driver uses parent bus directly)
  - GPIO aliases: reed-switch → GPIO4, mode-btn → GPIO0 (boot button),
    buzzer → GPIO5, led0 → GPIO48 (onboard RGB, gpio1 pin 16)
  - gpio_inputs node: reed_switch and mode_button as gpio-keys
  - gpio_outputs node: buzzer and status_led as gpio-leds
  - SPI2 (OLED) and SPI3 (SD card) nodes commented out until hardware arrives
  - Flash partitions: NOT redefined — board DTS already has storage_partition
    at 0x3b0000 which NVS will use directly

ERRORS FIXED DURING BUILD BRING-UP
--------------------------------------------------------------------------------

1. DTC overlay not being applied
   Root cause: Zephyr 4.x qualified board targets (esp32s3_devkitc/esp32s3/procpu)
   don't auto-discover overlays named after just the board root name.
   Fix: Added set(DTC_OVERLAY_FILE "${CMAKE_SOURCE_DIR}/boards/esp32s3_devkitc.overlay")
   to CMakeLists.txt before find_package(Zephyr).

2. CONFIG_PREEMPT_ENABLED in prj.conf
   Not a user-configurable Kconfig symbol — removed.

3. Inline comments on Kconfig lines
   CONFIG_FOO=y  # comment is invalid Kconfig syntax — removed all inline comments.

4. storage_partition and slot0_partition label conflicts
   Our overlay defined these; the board DTS already defines them at different
   addresses. Removed our definitions — board DTS versions are used directly.

5. cs-gpios not in zephyr,sdhc-spi-slot binding
   The SPI SD card DT binding doesn't accept cs-gpios on the slot child node.
   Entire &spi3 node commented out until hardware arrives and binding is verified.

6. CONFIG_FAT_FILESYSTEM_ELM enabled without a disk driver
   FatFS requires a mounted disk. With CONFIG_FILE_SYSTEM=y but no SDMMC driver,
   it errors "Wrong FF_VOLUMES setting." Fix: commented out all filesystem
   Kconfig options until SD card hardware is wired.

7. ff.h: No such file or directory in sdlog.c
   ELM FatFS header only exists when CONFIG_FILE_SYSTEM is set.
   Fix: wrapped entire sdlog.c FatFS implementation with #ifdef CONFIG_FILE_SYSTEM
   and added stub fallbacks returning -ENOTSUP in the #else branch.

8. APP_PRIO_LOGGER undeclared in alarm_service.c
   Missing #include "app_config.h" — added.

9. #include <zephyr/drivers/display.h> compile error
   Display subsystem disabled in prj.conf. Fix: guarded with #ifdef CONFIG_DISPLAY.

10. oled_font5x7.h missing
    File not created yet (Phase 2 TODO). Fix: removed include, replaced with
    a comment explaining what the font rendering stub needs.

11. CP210x USB-to-UART driver not installed on Windows
    ESP32-S3 uses a CP2102N USB bridge. Without the Silicon Labs driver,
    Windows does not create a COM port and west flash finds "0 serial ports."
    Fix: download and install CP210x Universal Windows Driver from silabs.com.
    (Pending — user needs to do this before first flash.)

THINGS THAT MAKE THIS PROJECT STAND OUT (for portfolio / video)
--------------------------------------------------------------------------------

- Written entirely for Zephyr RTOS — not Arduino, not ESP-IDF. Zephyr is used
  in products from Nordic Semiconductor, Intel, and dozens of IoT companies.

- Zero "magic numbers" — every constant lives in app_config.h with a comment
  explaining the value and what changes it affects.

- ISR discipline — the reed switch ISR does nothing except submit a work item.
  All event posting happens in the work handler at thread context.

- Custom IMU driver — not using a library. Reading ICM-20948 registers directly
  from the datasheet, including multi-bank register map and 14-byte burst reads.

- Compile-time hardware gating — if SD card or OLED isn't wired, the entire
  driver gracefully degrades to stubs returning -ENOTSUP. The firmware compiles
  and runs on partial hardware from day one.

- CSV event log to SD card — every state transition and fault is timestamped and
  logged. The log file can be opened in Excel for post-analysis.

- NVS-backed boot counter — the device remembers how many times it has booted
  and how many consecutive failures occurred, survives power cycles.

- Diagnostic mode — hold the boot button at power-on, get a full hardware self-
  test on the OLED: WHO_AM_I check, live IMU stream, reed state, SD test write.
